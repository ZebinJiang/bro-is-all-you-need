GenesisVLA 重构蓝图与执行计划

以 StarVLA 为母框架，吸收 FluxVLA 的 runner / deployment / acceleration 工程闭环，并引入 Dexbotic 的 dataclass config 与 transform pipeline 思想，形成一个研究灵活、工程严格、可大规模训练、可部署、可长期演化的 VLA 平台。

0. 执行摘要

最终命名：GenesisVLA。旧称 AsterForge 不再作为正式工程名使用；StarVLA 在新分支中作为上游母框架来源保留历史引用，但新代码、包路径、训练层级、部署层级、文档和 CLI 均统一迁移到 GenesisVLA。

GenesisVLA 的目标不是简单 fork FluxVLA，也不是在旧 StarVLA 代码上堆补丁，而是在 StarVLA 内部建立一个新内核：保留 StarVLA 的研究灵活性和低耦合框架边界，吸收 FluxVLA 在 FSDP runner、checkpoint lifecycle、remote inference、RTC、Triton/CUDA Graph acceleration 上的工程经验，再吸收 Dexbotic 的 dataclass config 与 transform pipeline 组织方式。

这条路线的战略优势是：我们借鉴 FluxVLA 的工程成熟度，但不继承它的 engine 绑定和历史结构；我们借鉴 Dexbotic 的可读配置体系，但不继承它的 BaseExp 上帝类、EasyDict 过渡债和 benchmark 脚本重复；我们保留 StarVLA 的研究基因，但通过严格类型系统、TDD、CI gate、runner 分层和部署层把它锻造成长期平台。

3–6 个月形成可用 GenesisVLA 分支
7 层架构core / config / data / model / runner / deploy / acceleration
TDD-first每个 milestone 先写 contract tests
兼容迁移旧 StarVLA 通过 legacy adapters 逐步迁移

1. Blueprint：GenesisVLA 的系统蓝图

1.1 一句话目标

GenesisVLA 是一个面向未来 VLA 研究与部署的工程内核，目标是把 StarVLA 的研究可塑性提升到 FluxVLA 级别的工程闭环，并以 Dexbotic 式 typed config 和 transform pipeline 保证可维护性。

1.2 三个来源如何融合

来源保留 / 吸收明确不继承
StarVLA 研究型低耦合框架、raw sample interface、framework 作为模型 API surface、benchmark 插拔能力、可单模块 smoke test 的文化。 弱类型全局 dict、全仓 lint backlog、pyright off、checkpoint state 不完整、训练脚本和 runner 耦合、当前 QwenGR00T 作为 GR00T-native 的替代品。
FluxVLA FSDPTrainRunner 思想、eval-after-train 生命周期、safetensors 优先、checkpoint/runner 工程闭环、ZMQ/remote inference、RTC、Triton/CUDA Graph acceleration 方向。 FluxVLA engine 绑定、过重 registry、GR00T 作为 LlavaVLA + name_mapping 的历史适配形态、任务配置过度承载系统结构。
Dexbotic dataclass config、experiment-centric 配置可读性、transform pipeline 组合思想、train backend enum、真实机器人文档组织思路。 BaseExp 上帝类、EasyDict 过渡债、benchmark 脚本重复、默认路径硬编码、把 pipeline 与 prompt/tokenization 过度耦合。

2. 为什么不是直接使用 FluxVLA 作为基座

FluxVLA 的工程闭环很强，它公开定位为 full-stack、end-to-end 的 embodied intelligence engineering platform，并明确强调 unified configuration、standardized interfaces、module decoupling 和 deployability。其 README 还显示已经支持 ZMQ-based remote inference、Pi0.5-RTC、Triton backend 等能力。

但直接以 FluxVLA 为基座会带来框架重心锁定：新增模型、数据、runner、eval 和 deployment 都自然进入 FluxVLA 的 engine、registry 和配置方式。对于“长期建立自有 VLA 平台”的目标，这会让架构主权转移到 FluxVLA。

核心判断：FluxVLA 是优秀参照物，不是理想母体。GenesisVLA 要吸收 FluxVLA 的工程生命周期，但不继承 FluxVLA 的所有 engine 绑定。
维度直接用 FluxVLAGenesisVLA 路线
短期跑通更快较慢，但架构更可控
架构主权受 Flux engine 影响完全 StarVLA/GenesisVLA-native
研究灵活性中等强
工程成熟度初始强通过 milestones 建设
长期可塑性中等强
新范式接入要适配 Flux runner/config可以定义自有 Protocol
风险被现有复杂度锁定建设成本较高

4. Repo 的清楚分级拆解

GenesisVLA 采用七层架构，每一层都必须拥有稳定输入、输出和测试边界。

层级职责不能做什么主要测试
Core类型、协议、registry、legacy adapters不能依赖模型、数据集、训练后端unit + static type
Configtyped config、loader、validation、export、migration不能把业务逻辑藏在 YAML 中schema + override + migration
Datadataset、transform、processor、statistics、collatordataloader 不做模型专属处理fixture + roundtrip + determinism
Modelframework、backbone、action head、policy模型层不管理训练分布式细节shape + backward + masked loss
Runnertrain/eval loop、distributed、checkpoint、callbacksrunner 不知道具体模型内部结构fake model train/resume
Deploymentpolicy server/client、schema、runtime、healthcheckdeployment 不读取训练态 optimizer/schedulerHTTP/ZMQ roundtrip
Accelerationtorch.compile、CUDA Graph、Triton、export、latency benchmark不能污染 reference eager patheager-vs-accelerated numerics

6. 关键接口与抽象

6.1 核心类型

@dataclass(frozen=True)
class RawSample:
    images: dict[str, ImageLike]
    language: str
    actions: np.ndarray | None
    state: np.ndarray | None
    robot_tag: str
    metadata: Mapping[str, Any] = field(default_factory=dict)

@dataclass(frozen=True)
class ActionChunk:
    values: np.ndarray | torch.Tensor
    mask: np.ndarray | torch.Tensor | None
    horizon: int
    action_dim: int
    normalized: bool

@dataclass(frozen=True)
class FrameworkOutput:
    loss: torch.Tensor | None
    losses: Mapping[str, torch.Tensor]
    metrics: Mapping[str, float]
    action_pred: ActionChunk | None = None

6.2 Protocols

class TransformProtocol(Protocol):
    def __call__(self, sample: RawSample) -> RawSample: ...

class FrameworkProtocol(Protocol):
    def forward(self, batch: ModelInput) -> FrameworkOutput: ...
    def predict_action(self, obs: PolicyObservation) -> ActionChunk: ...

class RunnerProtocol(Protocol):
    def setup(self) -> None: ...
    def train(self) -> RunnerState: ...
    def evaluate(self) -> dict[str, float]: ...
    def save_checkpoint(self, step: int) -> Path: ...
    def resume(self, path: Path) -> int: ...

class PolicyProtocol(Protocol):
    def reset(self) -> None: ...
    def select_action(self, observation: PolicyObservation) -> ActionChunk: ...

9.2 Python 规范

  • Formatter：Black。
  • Lint：Ruff。
  • Type：Pyright strict for new modules。
  • Docstring：Google style。
  • Line length：100。
  • 禁止 wildcard import。
  • 禁止 library code 中使用 print，统一 logger。
  • 禁止核心边界出现裸 dict[str, Any] 穿透。
  • 禁止 transform 内部隐式 device transfer。
  • 禁止隐式 .cuda()，统一使用 DeviceSpec 或 runner device。

9.3 Config 规范

  • YAML 是用户界面，dataclass 是内部真相。
  • 所有 config 必须可 export resolved。
  • 所有 config 必须记录 schema version。
  • breaking config change 必须提供 migration function。
  • 禁止在 benchmark script 中硬编码绝对路径。

9.4 Test 规范

  • 每个新 feature 必须包含 unit test、smoke test、failure-mode test、config validation test、minimal documentation snippet。
  • core/config/training/deployment 新目录覆盖率目标 ≥ 80%。
  • model/action_heads 覆盖率目标 ≥ 70%。
  • acceleration 允许 GPU optional，但 reference path 必须测试。

9.5 CI/CD 规范

PR CI:
  - lint new dirs
  - pyright strict new dirs
  - CPU unit tests
  - config schema tests
  - fake e2e smoke

Nightly CI:
  - 1 GPU smoke train
  - fake deployment server
  - tiny dataset train/eval
  - checkpoint resume
  - optional acceleration benchmark

Weekly CI:
  - multi-GPU FSDP smoke
  - one real benchmark subset
  - memory regression report

12. 风险矩阵

旧 benchmark 无法迁移中中legacy adapter 保留。
