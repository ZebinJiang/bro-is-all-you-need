HARD BOUNDARIES FOR AUTOVLA / STARVLA ENGINEERING

1. This repository is scoped to one project: StarVLA-based AutoVLA engineering. The current engineering base is StarVLA, and the target project blueprint is AutoVLA. Do not introduce unrelated project governance or imply unsupported model implementations.
2. The Manager must inspect the actual StarVLA repository layout before choosing target paths. Do not force AutoVLA source into a template-owned `src/` tree.
3. Future platform source may use native locations such as `autovla/`, `models/`, `engines/`, `datasets/`, `transforms/`, `tokenizers/`, `ops/`, `configs/<family>`, and `scripts/` entrypoints when an implementation plan explicitly introduces them.
4. Do not modify files outside the project repository unless the user explicitly gives a one-time external path exception for a specific path and task. Do not modify sibling repositories, parent directories, shared system paths, global shell configs, cluster configs, or user home files outside the explicit exception.
5. A one-time external path exception allows only the exact transfer or inspection described by the user. Record source, destination, command, size estimate, and evidence. After the task, do not touch that path again without a new explicit user instruction.
6. Do not mutate `code-input/`, `related-assets/`, `datasets/readonly/`, or `assets/input/` unless the user explicitly asks. The template guidance files `code-input/README.md` and `related-assets/README.md` may be updated by approved governance migrations.
7. Preserve original dataset source files under `datasets/readonly/`. Derived copies, patched data, indexes, manifests, converted datasets, and feature caches must use `datasets/working/` or `datasets/cache/`.
8. Do not copy full datasets into every `runs/` directory. Runs may store logs, manifests, metrics, small samples, symlinks, references, or checksums.
9. Keep generated logs, checkpoints, Slurm outputs, and experiment artifacts under `runs/` unless the user authorizes another project-local path or a one-time external storage transfer. Ad-hoc Manager/subagent temporary files that are not part of a specific run belong under `runs/tmp/`; run-specific temp remains under `runs/local/<run_id>/tmp`, `runs/slurm/<run_id>/tmp`, or `runs/slurm_debug/<run_id>/tmp`. Do not write temp files to the repository root; avoid system `/tmp` except when a tool cannot be redirected, and record that exception if it matters.
10. Do not commit generated run outputs, logs, checkpoints, cache files, secrets, model weights, or large datasets.
11. Do not use production secrets, credentials, certificates, tokens, private endpoints, robot endpoints, or paid third-party services unless explicitly authorized.
12. Do not perform destructive database operations, destructive filesystem cleanup outside this repository, or infrastructure/deployment changes unless explicitly requested.
13. Cleanup requires proposal first. A subagent proposes candidate deletions; the Manager audits; the user explicitly confirms; only then may deletion occur.
14. Do not modify Slurm cluster configuration, accounting, partitions, QoS, cgroups, prolog/epilog scripts, modules, or scheduler policy.
15. Do not cancel, reprioritize, inspect beyond permitted metadata, or interfere with other users' Slurm jobs.
16. Do not bypass Slurm scheduling, Slurm cgroups, GPU/CPU/memory limits, accounting, or site policy. Do not run unmanaged long workloads on login nodes.
17. Login nodes may run only lightweight repository-structure checks. Debug, tests, evaluation, training, inference serving, dataset conversion, and other compute-heavy work must run on compute nodes via project Slurm helpers or formal `sbatch` submission when they need real runtime resources.
18. If active Slurm config contains `TO_FILL`, run Slurm environment discovery and fill the config before formal submission. Once filled, do not update it unless the user explicitly requests an update.
19. Manager/agents must use project Slurm wrappers/helpers for project work. The wrappers must log equivalent raw `sbatch`/`srun` commands for user reproducibility. Raw direct commands are only allowed as a recorded one-off user exception or for the user manually outside agent execution.
20. Do not mark Slurm-dependent work complete without real submitted job evidence unless the feature is explicitly local-only or dry-run-only.
21. Every new task must be developed on a `dev/*` branch. Do not develop task changes directly on `main` or `master`.
22. The Manager may open a PR after a task. The Manager may merge only when the user explicitly authorizes Manager-side PR review and merge.
23. If a previous PR was merged, sync or recreate the dev branch before new work.
24. Any design/architecture implementation, code modification, optimization, debugging/fix implementation, structural/source/module/script/test/config-contract/model-path/performance/Slurm-wrapper/dataset execution, or inference execution change must be implemented by exactly one write-capable task-specific subagent by default, without waiting for an explicit user request for subagents. Structural/design/code/debug/optimization changes are serial; the Manager scopes/delegates, reviews, validates, records evidence/risks/rollback notes, and decides completion state, but must not directly implement those changes.
25. Read-only review, test analysis, related-assets analysis, daily-task planning, and cleanup proposal may run in parallel if they do not concurrently edit source or shared contracts.
26. Subagents may not set `passes: true`, mark tasks accepted, merge PRs, or make final milestone decisions.
27. Use the superpowers planning skill when related papers, notes, or open-source references under `related-assets/` drive implementation.
28. Use superpowers-style clarification for ordinary operational tasks involving cleanup, dataset placement, git history, storage transfer, or execution-state changes.
29. All registered VLA baselines are protected. Direct baseline-path edits require explicit task scope, rationale, validation evidence, and rollback notes.
30. A registered baseline family name is governance vocabulary only until source, config, checkpoints, datasets, and validation evidence are registered. Do not imply implementation merely because a family is listed.
31. Prefer minimal, reversible changes. Avoid broad refactors and avoid deep wrapper stacks unless the feature list requires them.
32. New or modified code comments and docstrings must be Chinese.
33. Manager must assess time complexity, space complexity, data movement, GPU/distributed efficiency, and baseline contamination risk before accepting subagent output.
34. Governance overlay files and local skills remain local-only unless the user explicitly asks to version or publish them. Do not add local governance overlay files to the StarVLA base history by default.
35. Do not synchronize StarVLA upstream branches unless the user explicitly asks for sync. When sync is requested, preserve local governance overlay files and record conflicts, commands, and validation.
36. Agent-created or agent-modified datasets, checkpoints, caches, logs, outputs, temporary files, and validation evidence must use `datasets/` and `runs/`. StarVLA-native `playground/`, `results/`, or similar paths may not be used for agent work unless rewritten or mapped into governed paths for that task.
37. Hugging Face downloads are allowed when task-relevant and must land in governed paths. Hugging Face uploads are forbidden. W&B logging is allowed only under project `zjh-flywheel`; service tokens and API keys must never be written to repository files or commits.
38. Real robot endpoints, policy servers, remote endpoints, and action-producing clients require explicit task authorization and an allowed action scope. Record target, command, safety checks, logs, outputs, risks, and rollback/stop procedure.
39. Do not modify global Python, base Conda, system packages, shared modules, or user shell configuration. Dependencies may be installed only into a project venv, project-specific Conda env, container, or user-specified one-time environment with recorded commands and validation.
40. Third-party code may be integrated for local experiments only after recording source, license status, purpose, and risk. Any copied external code must carry a file-header attribution. Do not publish or push externally sourced code with unresolved license risk.
41. Before every commit, push, or PR, run secret scanning and large-file scanning. Block credentials, API keys, private endpoints, model weights, checkpoints, dataset dumps, large run outputs, and unauthorized service config unless the user explicitly overrides and the override risk is recorded.
42. The repository rules are standing user authorization to use task-specific subagents for structural implementation. Parallel subagents are allowed only for independent tasks with disjoint write paths and no shared contract, baseline, dataset execution chain, or Slurm-wrapper conflict.
43. Local commits on `dev/*` are allowed after validation and scans. Do not push, create PRs, publish, or update remote branches unless the user explicitly asks. Use the user-provided GitHub proxy for GitHub network operations when needed.
